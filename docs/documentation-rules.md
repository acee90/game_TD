# 문서화 규칙 — v0.2

> 상태: **현행** · 최종 갱신: 2026-07-10
> 이 저장소의 문서를 쓰고 고치는 사람(사람이든 에이전트든)이 따르는 규칙.
> 핵심 전제: **게임 디자인 문서는 계획서가 아니라 결정 기록이다.**
> 플레이해보기 전엔 재미를 알 수 없으므로 문서가 구현을 앞서지 않는다 —
> 프로토타입 → 측정 → 결정 → 기록의 순서로 쓴다.
>
> v0.2: 외부 리뷰 반영 — 비전·필러 실문서화, 상태 메타데이터, 정량/정성 근거 구분,
> 버전 정책 분리, 목표 지표.

## 2. 모든 문서에 상태 헤더를 단다

문서 상단에 한 줄:

```
> 상태: 현행 | 초안 | 대체됨(→ 대체 문서 링크) | 보관 · 최종 갱신: YYYY-MM-DD
```

- **현행** — 지금 유효한 기준.
- **초안** — 논의 중, 아직 기준 아님.
- **대체됨** — 다른 문서가 역할을 이어받음. 반드시 링크.
- **보관** — 역사적 기록. 참고는 하되 기준으로 쓰지 않는다.

새로 온 사람(과 에이전트)은 상태가 "현행"인 문서만 기준으로 삼는다.

## 3. 수치의 원본은 코드다

```
engine/src/data/*.ts ← 유일한 원본. 수치는 여기만 고친다
   ↓ npm run gen:balance
docs/balance/*.csv  ← 자동 생성. 손으로 고치지 않는다
   ↓ 업로드 (요청 시)
구글 시트            ← 열람·공유용 읽기 전용 사본
```

- `engine/tests/balance-csv.test.ts`가 코드-시트 일치를 강제한다.
- 역방향(시트 → 코드) 동기화는 하지 않는다.
- 이 방침은 밸런스를 만지는 사람 = 코드를 만지는 사람인 동안 유효하다.
  비개발 기획자가 합류하면 "시트가 원본, 게임이 import" 모델로 전환을 검토한다.
- 문서 본문의 수치는 스냅샷이다. 문서와 코드가 다르면 코드가 맞다.

## 4. 근거 — 정량과 정성 모두 인정한다

**근거 없는 수치 조정을 금지한다.** 근거는 두 종류이며 서로 대체하지 못한다:

- **정량 (시뮬레이션)** — 시드 봇, GA, 파워 곡선. "이길 수 있는가 / 선택이 성립하는가"를
  검증한다. 주장에는 시드 수와 재현 방법을 붙인다.
- **정성 (플레이테스트)** — 관찰, 인터뷰, 행동 기록. "무엇을 해야 하는지 이해했는가 /
  실패 이유를 알았는가 / 지루하거나 부당하게 느꼈는가"를 검증한다.
  시뮬레이터는 이것을 못 본다.

재미와 가독성 판단에는 정성 근거가 필수다. 목표 지표(명세 §목표 지표)에 정량 지표와
함께 경험 지표도 둔다.

## 5. 명세와 백로그를 섞지 않는다

- 명세에는 **구현·결정된 것만**.
- 미확정 아이디어는 [backlog.md](design/backlog.md)로. 채택되면 명세로 옮기고,
  기각되면 사유와 함께 백로그의 "기각" 절로.
- 명세의 "미해결 과제"는 백로그가 아니라 **측정으로 확인된 알려진 문제**의 목록이다.
- 실험 중인 변경은 백로그에 "실험 중" 표기로 두고, 검증이 끝나면 명세의 결정 기록으로.

## 6. 설계 결정 기록 (ADR)

시스템을 바꾸는 결정마다 명세의 "설계 결정 기록" 표에 추가한다:

| 결정 | 이유 | 상태 | 검증 |
|---|---|---|---|
| 무엇을 | 왜 (기각 대안 포함) | 채택/폐기/대체됨 | 근거 리포트·테스트 링크 |

날짜는 git이 갖고 있으므로 적지 않는다. 같은 논쟁이 주기적으로 돌아오므로
"왜 그렇게 안 했는지"가 "왜 그렇게 했는지"만큼 중요하다.

## 7. 버전 정책 — 문서 종류마다 다르다

| 종류 | 정책 |
|---|---|
| 비전·필러, 명세, 규칙, 백로그 | **고정 파일명** + 최신 상태 유지. 버전은 문서 안에 표기. 이력은 git |
| 측정 리포트 | **불변 스냅샷.** 새 실행 = 새 버전 파일 또는 명시적 vN 섹션 추가. 기존 내용 수정 금지 |
| 자동 생성 CSV | 무버전, 항상 현재값 |

측정 리포트 상단에 기준 커밋 해시를 적는다 (`커밋 e75e246 기준`).

## 8. 파일명과 등록

- 파일명은 **영문 kebab-case** (`game-system-spec.md`). 제목·본문은 한국어.
- 폴더는 계층을 따른다:

```
docs/
├── design/      기준 문서 — vision / game-system-spec / backlog
├── reports/     측정 리포트 (불변 스냅샷) · 플레이테스트 기록
├── reference/   원본 분석 자료
├── archive/     대체된 문서
└── balance/     자동 생성 CSV
```

- 새 문서를 만들면: (1) 위 폴더 중 어디인지 정한다 — 애매하면 만들지 말고 기존 문서에
  절을 추가한다. (2) [docs/README.md](README.md) 문서 지도에 상태와 함께 등록한다.
  (3) 측정 기반이면 재현 방법(시드, 명령어)을 문서 끝에 적는다.
