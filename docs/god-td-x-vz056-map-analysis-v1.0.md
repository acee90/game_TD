### 0.2 추출 절차

1. scmscx API `/api/uiv2/map_info/{id}` → `mpq_hash` → `/api/maps/{hash}`로 `.scx` 다운로드.
2. `.scx`는 MPQ 아카이브이며 내부가 PKWARE implode 압축이라 mpyq로는 풀리지 않는다. StormLib(`brew install stormlib`)를 ctypes로 호출하여 `staredit\scenario.chk`를 추출.
3. CHK 섹션(DIM/ERA/MTXM/MRGN/UNIT/UNIx/UPGx/PUPx/STRx/TRIG/OWNR/SIDE/FORC)을 직접 파싱.
4. 문자열 인코딩은 **UTF-8**(cp949 아님).
5. 파싱 도구는 저장소 `tools/chk/`에 커밋되어 재현 가능.

### 0.3 EUD 한계 (정확한 수치)

이 맵은 **EUD 맵**이다(`get_death_euds=285`, `set_death_euds=409`). 맵 로직의 대부분이 컴파일된 EUD 코드로 구현되어 있어 일반 트리거로 읽을 수 없다.

- **STRx 섹션 13,625,656 바이트 중 실제 문자열은 19,699 바이트(1,025개)뿐이고, 나머지 13,605,957 바이트(99.9%)는 문자열 테이블 뒤에 붙은 EUD 페이로드다.**
- 트리거는 총 **866개**. 각 트리거는 조건 16칸×20B + 액션 64칸×32B 고정 구조이며, 종료 슬롯(type=0) 뒤 공백에 EUD 데이터가 채워져 있어 첫 종료 슬롯에서 잘라 읽었다. 그 결과 **유효 액션 3,501건, 유효 조건 855건**. (자르지 않으면 CreateUnit 94→192건, Victory 0→54건으로 존재하지 않는 로직이 부풀려진다.)
- 866개 중 **704개는 전부 유효 opcode로 해독 가능**, 162개는 범위 밖 opcode(EUD 데이터)를 포함한다. 유효 opcode는 액션 0~59, 조건 0~23뿐이다.
- SetDeaths 액션 2,149건 중 약 442건은 EUD 메모리 쓰기(플레이어 번호 >11 또는 값 >100,000), 나머지는 게임 상태 변수(카운터).
- **`Order` 액션 0건, `Victory`/`Defeat` 액션 0건.** 즉 적군 이동 경로와 승패 판정은 일반 트리거에 없고 EUD 계층에 있다.

### 0.4 이 문서에 "없는" 것 (EUD로 인해 판독 불가)

- 미션 완료 판정·보상의 구체 수치

## 4. 유닛 로스터

### 4.1 종족 체계

## 5. 조합 시스템

### 5.1 2단계 파이프라인

**1단계(조합 버튼 트리거):** 같은 유닛 2기 + 조합버튼(unit#83)이 플레이어 존에 모이면, 2기를 loc2로 MoveUnit해 소비하고 티어별 '요청 카운터'(death 변수)를 +1 한다. 근거: trigger #207(메딕) — Bring(CurrentPlayer,[Lv1]메딕,AtLeast,2)+Bring(CurrentPlayer,unit#83,loc153,AtLeast,1) → MoveUnit([Lv1]메딕 x1, loc155→loc2)+SetDeaths(CurrentPlayer,unit#177,Add,1).

**2단계(생성 트리거):** 요청 카운터≥1 이고 선택자값이 특정 N일 때 해당 티어 유닛 1기를 loc5(+Lv4/GOD는 loc6 토큰도)에 생성. 근거: trigger #496 — Deaths(CurrentPlayer,unit#177,AtLeast,1)+Deaths(P6,unit#73,Exactly,1) → SetDeaths(unit#177,Subtract,1)+CreateUnit([Lv2]어사돈 x1, loc5).

- **요구 수량 = 2**(x2, x3 아님). 근거: trigger #207/#209/#258/#260 전부 AtLeast 2. UI: strings:296 "조합하려면 유닛 2개가 필요합니다."
- **조합에 트리거상 미네랄/가스 차감 없음**(8절). 비용은 소모되는 입력 유닛 2개(네이티브)일 뿐.

티어 내 '어느 유닛'은 랜덤이 아니라 공유 인덱스 **unit#73**(`Deaths(P6, unit#73, Exactly, N)`)으로 결정된다. unit#73은 문자열상 '랜덤뎃값'(strings:357)이나, 판독 가능한 범위에서 이를 세팅하는 것은 보스 처치 시 +1(trigger #602~#606)뿐이다. 매 프레임 1~7을 도는 랜덤화·리셋 트리거는 EUD로 판독 불가(11절). 따라서 "조합 결과가 랜덤"이라는 통설은 판독 가능 범위에선 **진행/카운터 기반**이며, 순수 랜덤 여부는 미확인.

## 6. 갓 시스템

### 6.1 갓 변환(스플↔파워 상호 전환)은 이 버전에 없음

'조합' 트리거들(#440~#491 등)은 loc6에 유닛이 Bring되면 '<색> 플레이어가' + '<유닛명> 【태그】를 조합하였습니다.' 방송 후 MinimapPing + KillUnitAt로 그 유닛을 죽이는 **알림+소거**일 뿐이며, 한 [GOD] 유닛을 다른 [GOD] 유닛으로 바꾸는 트리거는 존재하지 않는다. strings/trig_chunks 전체에서 '전환'/'변환' 문자열 0건.

> **커뮤니티 통설 "첫 갓은 스플로 변환한다" 등의 태그 토글 메커니즘은 이 맵 데이터로 확인되지 않는다.** 각 유닛의 【태그】는 이름마다 고정(복합 태그도 고정 세트)이며, 한 유닛이 서로 다른 태그로 두 번 등장하지 않는다.

가스는 프로브 채취(네이티브)로 얻고 업그레이드에 쓰인다고 TIP에 명시(strings:11 '업그레이드는 파일런에서', strings:13 '프로브를 뽑아 가스를 채취'). 그러나 **트리거 내 Upgrade/SetUpgrade 액션 0건, 가스 차감 트리거 0건.** 업그레이드는 파일런의 네이티브 SC 업그레이드로 처리되는 것으로 보이나 트리거 근거는 미확인. ('Pylon' 언급은 전부 SetDeaths 점수 카운터.)

### 8.5 upgrades.txt

15개 업그레이드(ID 46~60) **전부 usesDefault=1**, baseMin/factorMin/baseGas/factorGas/baseTime/factorTime = **전부 0**. 커스텀 비용표 없음(SC 기본 비용 적용). p1maxLvl/p1startLvl은 SC 정상 상한을 벗어난 비정상값(예: upg46 maxLvl=228/start=52, upg47 236/131, upg59 247/143)을 다수 포함해 게임 최대레벨로 해석하기 어렵다. usesDefault=1이라 실제 반영·의미는 미확인.

### 8.6 P7 가스 = 라운드 카운터

P7(컴퓨터)의 `Accumulate(...,gas)` 조건들은 EUD가 아니라 정상 범위 카운터다(값 1,3,7,22,64 등 소수). P7 gas에 웨이브/레벨 카운터가 저장되어 스폰을 분기한다(9절). 이는 인간 플레이어의 자원 경제가 아니라 게임 진행 카운터.

## 12. 우리 게임(랜덤합성 TD)에 대한 시사점

> **이 절만 해석·의견이 허용된다.** 1~11절 사실을 근거로 한 설계 판단이다.

### 12.1 검증된 뼈대: "2개→1개 결정적 인덱스 합성 + 방송 이벤트"

이 맵의 합성은 통설과 달리 **완전 랜덤이 아니라 공유 인덱스(unit#73)로 결정되는 준결정적 구조**로 판독되었다(5.3). 우리 게임이 '랜덤합성'을 표방하더라도, 실제로 재현 가능하고 디버깅 가능한 시스템을 원한다면 이 맵처럼 **명시적 카운터/시드 기반 선택**을 두는 편이 낫다. 순수 RNG로 보이게 하되 내부는 결정적 인덱스로 굴리면, 밸런싱·QA·리플레이가 훨씬 쉬워진다.

### 12.2 티어 사다리는 "요청 카운터"로 느슨하게 결합하라

이 맵은 입력 티어마다 별도 요청 카운터(unit#177/#220/#221)를 두고, 생성 단계를 카운터 감산으로 분리했다(5.1~5.2). 이 소비자-생산자 분리 패턴은 우리 게임에서도 **합성 요청과 결과 생성을 분리**하는 이벤트 큐로 구현할 가치가 있다. 다만 이 맵의 Lv3 rung이 EUD로 잘려 우리가 검증조차 못 한 점(11.2)은 반면교사다 — **합성 파이프라인은 전 티어가 동일한 판독 가능 구조**여야 유지보수가 된다.

### 12.3 경제는 "지출 없는 킬 보상 누적형"이 핵심 축이었다

플레이어 자원 차감이 사실상 없고(8.3), 소득은 보스 처치·킬 마일스톤·반복 20킬 3계열의 **Add 누적**뿐이었다(8.2). 즉 이 장르의 재미는 '자원 관리'보다 **킬 스코어를 끝없이 쌓는 성장 곡선**에 있다. 우리 게임에서도 마일스톤 보상 테이블(200킬 간격, +5~+15의 완만한 증가)은 그대로 참고할 만한 튜닝 앵커다. 단, 이 맵의 보상 수치가 3231배수처럼 보이다 Lv6에서 깨지는 것(7.2)은 **하드코딩 수치의 관리 실패 사례**다 — 우리는 공식/데이터 테이블로 관리하자.

### 12.4 태그(파워/스플래시/스피드)는 유닛 고정 속성으로 설계하라

'상호 전환'은 이 맵에 없었다(6.1, 11.5). 대신 유닛마다 태그가 고정이고 GOD은 대체로 2속성 복합이었다(4.3). 우리가 '전환' 메커니즘을 넣고 싶다면 그건 **이 맵을 넘어서는 신규 설계**임을 명확히 인지해야 한다 — 원작 재현이 아니다. 원작 감성을 원하면 태그는 고정으로 두고, 대신 **합성 결과 유닛의 태그 조합(단일/복합)으로 티어감**을 표현하는 편이 원작 충실하다.

### 12.5 대칭 십자맵 + 4인 개별 arm은 검증된 레이아웃

중심 (61,54) 기준 완전 대칭 십자 스폰(9.1)과 arm별 플레이어 격리(2.3)는 4인 경쟁형 TD의 안정적 골격이다. 스폰이 웨이브마다 사용 방위를 달리한 점(9.2)은 단순 반복을 피하는 좋은 장치다. 다만 **이동 경로 자체는 이 맵이 EUD로 숨겨** 우리가 재현할 근거가 없다(9.5, 11.1) — 우리 게임은 경로를 **명시적 웨이포인트 데이터**로 두어 튜닝·시각화가 가능하게 하는 것이 이 맵보다 나은 선택이다.

### 12.6 EUD 의존은 '분석 불가능한 게임'을 만든다 — 우리는 반대로

이 맵 로직의 99.9%가 EUD 페이로드였고(0.3), 그 결과 경로·승패·미션·난이도·쿨타임 등 핵심 시스템 대부분을 원본에서 읽어낼 수 없었다(11.1). 이는 스타 유즈맵의 기술적 한계를 우회한 위업이지만, **재현성·분석성·유지보수성 관점에서는 최악**이다. 우리 게임은 동일한 게임플레이 깊이를 **데이터 주도(data-driven) 테이블 + 결정적 로직**으로 구현해, 밸런싱과 QA가 파일만 보고도 가능하도록 설계하는 것이 이 문서 전체가 주는 가장 큰 교훈이다.
