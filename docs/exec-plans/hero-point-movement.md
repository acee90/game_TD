# 실행 계획 — 영웅 클릭 지점 이동과 경로 피드백

> 상태: **구현됨** (M0~M4 완료 — §7 결과. M5 사람 플레이테스트 대기.
> 코너 블렌딩은 [프로토 가설] — 검증 전이라 명세 미기록) · 최초 작성: 2026-07-16 · 최종 갱신: 2026-07-16
> 근거: [Unity HUD 플레이테스트 F6](../reports/unity-hud-playtest-v0.1.md)
> 범위: `web/src` 영웅 위치 모델 → Unity Core 미러 → Unity View 목적지·경로 연출

## 1. 문제

현행 영웅 위치는 경로 위 진행도 `Distance` 하나다. `X/Y`는 경로 중앙선에서 계산되고
`MoveTo(x, y)`도 클릭 좌표를 `NearestPathDistance`로 투영한다. 따라서 길 가장자리를 클릭해도
영웅은 중앙선으로 이동한다. Unity View만 임의로 옮기면 전투·스킬·몸막기 판정과 보이는 위치가
어긋난다.

## 2. 목표와 비목표

**목표**

- 영웅이 길의 보행 가능 영역 안에서 실제 클릭 지점까지 이동한다.
- 우클릭 즉시 실제 목적지와 예정 경로를 표시한다.
- 이동 속도, 사거리, 몸막기, 스킬 판정과 시드 결정성을 보존한다.

**비목표**

- 완전 자유 이동, NavMesh, 타워 사이 길찾기, 새 장애물 규칙을 도입하지 않는다.
- 적 경로와 타워 배치 규칙을 바꾸지 않는다.
- HUD F1~F5 작업과 한 커밋에 섞지 않는다.

## 3. 프로토타입 결정안 — 진행도 + 횡방향 오프셋

현재 경로 진행도에 횡방향 오프셋을 추가한다. 클릭 좌표를 가장 가까운 경로 구간의 진행도와 법선
오프셋으로 분해하고, 오프셋을 길 타일의 보행 가능 폭 안으로 제한한다. 이 방식은 기존 전후 진행과
몸막기 의미를 유지하면서 중앙선 고정만 제거한다.

1. 영웅에 현재/목표 횡오프셋을 추가한다.
2. 영웅 `X/Y`는 `PathPos(Distance) + PathNormal(Distance) * LateralOffset`으로 계산한다.
3. 맵 투영 함수가 최근접 진행도, 구간 법선, 제한된 횡오프셋, 실제 목적지를 함께 반환한다.
4. 이동은 진행도와 횡오프셋의 2D 거리로 속도 예산을 나눠 대각 이동 가속을 막는다.
5. 보행 영역 밖 클릭은 최근접 유효 지점으로 보정하고 목적지 마커도 보정된 위치에 둔다.

이 안은 아직 **프로토타입 가설**이다. 90도 코너에서 법선이 급변해 영웅이 튀면 코너 블렌딩 또는
짧은 2D waypoint 경로로 대체하고, 검증 전에는 명세에 확정 규칙으로 기록하지 않는다.

## 4. 구현 단계

### M0. 기준 테스트 · 규모 S

- 직선·코너·제단에서 현재 중앙선 투영 결과를 테스트로 고정한다.
- 영웅 사거리, 몸막기, 범위 스킬, 부활 위치 시드 회귀 기준을 만든다.
- **게이트:** 기준 테스트가 현행 코드에서 재현된다.

### M1. 웹 위치 모델 프로토타입 · 규모 M

- `web/src` 맵 투영과 Hero 모델에 횡오프셋을 추가한다.
- 직선, 코너, 범위 밖 클릭, 역방향 이동을 단위 테스트한다.
- **게이트:** 클릭 목표 오차가 게임 좌표 1px 이하이고 대각 이동 속도가 상수를 넘지 않는다.

### M2. 전투 회귀 · 규모 M

- 실제 `Game` 시뮬레이션으로 영웅 평타·범위 스킬·몸막기·부활을 검증한다.
- 횡오프셋이 적 경로 진행도 판정과 충돌하는 지점을 기록한다.
- **게이트:** 기존 시드 결과의 의도하지 않은 승패 변화가 없고 위치와 판정이 일치한다.

### M3. Unity Core 미러 · 규모 M

- 검증된 TypeScript 로직을 `Hero.cs`와 `MapData.cs`에 동일하게 옮긴다.
- 웹/Unity 양쪽에 같은 투영 fixture를 두어 결과를 비교한다.
- **게이트:** fixture의 진행도·오프셋·목적지 좌표가 허용 오차 안에서 일치한다.

### M4. 목적지·경로 연출 · 규모 M

- `GameView`가 우클릭 직후 유효 목적지 decal을 표시한다.
- 현재 위치부터 목적지까지 경로 구간을 샘플한 점선 또는 얇은 리본을 그린다.
- 새 명령, 도착, 사망, 부활 때 이전 표시를 정리한다.
- **게이트:** 표시 경로와 실제 궤적의 육안 불일치가 없고 HUD 클릭에는 표시가 생기지 않는다.

### M5. 플레이테스트 · 규모 S

- 직선 중앙/가장자리, 90도 코너, 범위 밖을 반복 클릭한다.
- 이동이 덜 허접해 보이는지, 보정 목적지를 이해하는지 관찰한다.
- 실패 결과는 새 플레이테스트 리포트에 기록한다.

## 6. 완료 기준

- 유효 지점 클릭 오차가 Unity 월드 0.1 이하이다.
- 90도 코너에서 경로 이탈·순간이동이 없다.
- 이동 속도가 방향에 따라 빨라지지 않는다.
- 목적지·경로 표시와 실제 궤적이 일치한다.
- 사거리, 몸막기, 보스 통과, 범위 스킬, 제단 부활, 시드 결정성 회귀가 없다.
- `web/src`와 Unity Core가 같은 입력에 같은 투영 결과를 낸다.

## 7. 구현 결과 (2026-07-16, M0~M4)

- **모델**: §3 결정안 그대로 — `projectToPath`(구간별 수선의 발, 해석적)가 클릭을
  (진행도, 횡오프셋)으로 분해하고, 횡은 보행 반폭(`WALKABLE_HALF_WIDTH` 12px)으로 클램프.
  이동은 (진행도, 횡) 2D 벡터 예산으로 속도를 나눠 대각 가속 없음. 90도 코너는
  **코너 블렌딩**(`CORNER_BLEND` 18px 안에서 횡을 선형으로 0에 수렴)으로 좌표 연속 —
  영웅이 코너에서 중앙선으로 모였다가 다시 벌어진다. [프로토 가설 — M5로 판정]
- **부수 변경**: `nearestPathDistance`가 600스텝 샘플링 → 해석적 투영으로 바뀌었다.
  의미는 동일하되 더 정확하다 (제단 진행도 등 미세 이동 가능성 — 전체 테스트 통과로 확인).
- **M0~M2 (웹)**: `tests/hero-movement.test.ts` 11개 — 중앙선 회귀 고정 · 가장자리 클릭
  오차 ≤1px · 폭 밖 보정 · 대각 속도 상한 · 코너 무순간이동 · 부활 리셋 · fixture.
  전체 291 테스트 통과 (기존 전투·보스·시뮬 회귀 없음 — 봇은 lateral을 안 쓴다).
- **M3 (Unity Core)**: `MapData.ProjectToPath`/`PathPosLateral` + `Hero.Lateral` 미러.
  fixture 4점((180,100)·(40,250)·(248,432)·중앙) 모두 웹과 소수 3자리까지 일치,
  `ALTAR_PATH_DISTANCE`도 184.000으로 동일.
- **M4 (Unity View)**: 우클릭 → 보정된 목적지 decal(파란 디스크) + 클릭 링 +

  도착(2D 엡실론)·사망·재시작 시 자동 정리. 플레이 모드 실측: 88px/s 정속,
  횡 보간 동작, 런타임 에러 0, 캡처로 마커·링 확인.
- **M5 (사람 플레이테스트)**: 대기 — 코너 블렌딩 체감(튀는가), 가장자리 이동의
