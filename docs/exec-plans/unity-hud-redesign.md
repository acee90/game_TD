## 0. 문제 — 오버레이가 보드를 삼면에서 포위한다

`GameHud.cs`의 패널 4개는 전부 **고정 픽셀**이다. 화면이 작아져도 줄지 않는다.

| 해상도 | topBar | rightPanel | heroPanel | messageBar | **합계** |
|---|---|---|---|---|---|
| 1920×1080 | 2.6% | 5.3% | 4.1% | 5.7% | **17.8%** |
| 1280×720 | 5.9% | 12.0% | 9.3% | 6.1% | **33.3%** |
| 958×385 (실측) | 14.8% | 30.0% ⚠ | 23.1% | 7.0% | **74.9%** |

⚠ 958×385에서 `rightPanel`(`y=12..498`)은 **화면 아래로 넘친다**(`GameHud.cs:129`).
`heroPanel`도 화면 높이를 초과한다(`:188`).

**근본 원인:** 웹은 캔버스(420×470) *옆에* 컨트롤이 세로로 흐르는 문서형 레이아웃이라
게임 화면을 전혀 가리지 않는다(`web/index.html:109-200`). Unity 포팅이 그 컨트롤을 그대로
**화면 위 오버레이로 옮기면서** 보드가 좌·하·우 삼면에 갇혔다.

**두 번째 문제 — 정보가 상시 노출된다.** 지금은 영웅 스탯·보스 6버튼·종족 4버튼·스탯 3버튼이
항상 떠 있다. 플레이어가 지금 무엇을 하려는지와 무관하게 전부 보인다.
RTS는 이 문제를 **선택 컨텍스트**로 푼다 — 고른 대상의 정보와 명령만 보여준다.

## 1. 설계 — 커맨드 패널, 단 TD에 맞게 깎아서

### 1.0 레퍼런스 조사가 교정한 것

SC 골격을 그대로 옮기려던 첫 초안을 조사가 정면으로 교정했다. 요약:

- **SC식은 우리 게임에 과하다.** "선택한 것의 모든 명령을 하단에 나열"하는 구조는
  수십 유닛을 미시조작하는 RTS에 특화됐다. 우리 조작 대상은 타워 41칸 + 영웅 1기뿐이다.
- **가장 가까운 레퍼런스는 Legion TD 2다.** 개발팀이 명시적으로 *"인터페이스 점유를 최소화하고
  하단 바 네비게이션 의존을 제거"*를 목표로 삼았다. 전역 단축키로 어디서든 조작하게 만들고 바를 얇게 유지한다.
  ([Legion TD 2 — First Look at the HUD](https://beta.legiontd2.com/updates/first-look-at-the-hud/))
- **WC3조차 커맨드 카드를 비웠다.** 이동·정지·공격·패트롤·홀드를 화면에서 **숨기고** 단축키로만 남겨,
  유닛 고유 명령 7개만 노출한다. ([Hive Workshop](https://www.hiveworkshop.com/threads/create-a-hero-with-attributes.167754/))
- **TFT는 커맨드 카드가 아예 없다.** 보드가 화면 대부분을 차지하고 하단은 골드·상점·벤치다.
- **그리드 단축키가 현대적이다.** SC 원작은 문자 단축키(M=Move)였고, 물리적 위치 기반(QWE/ASD/ZXC)은
  **SC2에서 도입**됐다. ([Blizzard — Basic Unit Controls](https://news.blizzard.com/en-us/article/4552956/game-guide-basic-unit-controls))
- **우클릭 = 맥락 명령은 Warcraft II(1995)가 대중화**했고 이후 표준이다. 학습 비용 0.

**우리 게임에 결정적인 사실: 영웅 스킬은 자동 시전이다.** 즉 커맨드 카드에 들어갈 "명령"이
사실상 없다. 들어가는 건 전부 **상점**(스탯 구매·스킬 개조)과 **전역 액션**(프로브·업그레이드·보스)이다.
→ **하단 바는 "명령 콘솔"이 아니라 "정보 + 상점"이다.** 이 재프레이밍이 바를 더 얇게 만든다.

**버린 것:**
- **미니맵** — 십자 41타일이 한 화면에 다 보인다. SC는 맵이 화면 밖에 있어서 필요했다.
- **다중 선택 와이어프레임 격자** — 타워는 격자 위치가 곧 정체성이다. 하단 중복 나열은 무의미.
- **이동·정지·공격 버튼** — 영웅은 경로를 자동 순회하고 스킬은 자동 시전이다. 명령 버튼 0개.
- **조합 진행도를 하단에 넣는 안** — 월드 오버레이(타일 위 아이콘)로 처리해 화면 가림을 피한다.

│                                                          │

위는 **영웅을 선택했을 때**다. 선택 대상이 바뀌면 세 열의 내용만 바뀌고 **구조는 고정**된다.
좌측에 **미니맵을 두지 않는다** — 십자 41타일이 한 화면에 다 보이므로 SC의 미니맵은 불필요하다(§1.0).

- **상단**: 자원·라운드·라이프·점수만. 지금의 `topBar`에서 보상 현황·보스 상태·피해 기여를 뺀다
  (각각 커맨드 카드 툴팁과 선택 정보로 이동).
- **하단 바 하나**로 통합. 지금의 `heroPanel` + `messageBar` + `rightPanel`을 전부 흡수한다.
- **보드 영역은 완전히 비운다.** 오버레이 0.

### 1.2 선택 모델 (신규)

현재 `Game.Selected`는 `Slot`만 가리킨다(`Game.cs:48`). 영웅을 선택한다는 개념이 없다.
**Core를 건드리지 않고** View에 선택 상태를 새로 둔다:

```
Selection = None | Hero | Tower(Slot) | EmptyTile(Slot)
```

- `Tower(slot)`을 고를 때 `Game.Selected`도 함께 갱신한다 — `SellSelected()`가 그걸 보기 때문
  (`Game.cs:284-289`).
- `Hero`·`EmptyTile`·`None`일 때 `Game.Selected = null`.

### 1.3 선택 대상별 하단 바

| 선택 | 선택 정보 (좌) | 상세 (중) | 커맨드 카드 (우) | 칸 |
|---|---|---|---|---|
| **영웅** | Lv 배지·HP바·XP바·부활 타이머 | 공격력·DPS·사거리·광역·피해감소·재생 / 증강 + 시너지 / **스킬 쿨타임 게이지** | 힘·민첩·지능 구매 · 스킬 피해+8% · 쿨타임-6% | 5 |
| **타워** | 종족색 이름·티어 라벨 | 공격력·간격·DPS·사거리·태그 | 판매 · *(GOD 한정: 타입 변경 — §4 보류)* | 1~2 |
| **빈 타일** | "빈 타일" | 생성 비용·현재 미네랄 | 유닛 생성 | 1 |
| **없음** | 게임 상태 요약 | 보상 현황(웨이브·마일스톤) · 피해 기여 | 프로브 · 종족 업그레이드 ×4 · **보스 소환 ▸** | 6 |
| └ **보스 하위 카드** | — | 다음 보스 보상 미리보기 | Lv1~Lv6 · Esc로 복귀 | 6 |

**스킬은 버튼이 아니다.** 자동 시전이므로 커맨드 카드가 아니라 **상세 열의 쿨타임 게이지**로 표시한다.

**전역 명령은 11칸이라 3×3을 넘친다** (프로브 1 + 종족 4 + 보스 6). WC3식 **하위 메뉴**로 접는다 —
보스를 한 칸으로 묶고, 누르면 카드가 Lv1~6으로 교체된다. Esc로 복귀.

**"선택 없음"이 기본 화면**이다. SC에서는 이때 카드가 비지만, 타워디펜스는 상시 전역 조작이
필요하므로 여기에 전역 액션을 둔다.

### 1.4 조작 체계 — 좌클릭 선택 / 우클릭 이동

지금은 좌클릭이 선택·생성·이동을 전부 겸한다(`GameView.cs:205-232`). SC 관용으로 가른다:

| 입력 | 지금 | 바뀜 |
|---|---|---|
| 좌클릭 — 빈 타일 | 유닛 **생성** | **선택** (생성은 커맨드 카드 버튼으로) |
| 좌클릭 — 타워 타일 | 선택 | 선택 (동일) |
| 좌클릭 — 영웅 | *불가* (콜라이더 없음, `GameView.cs:589`) | **선택** ← 콜라이더 추가 |
| 좌클릭 — 빈 곳 | 영웅 이동 | **선택 해제** |
| 우클릭 — 아무 곳 | *미사용* (`GetMouseButtonDown(0)`만 존재) | **영웅 이동** |
| 더블클릭 — 빈 타일 | — | 유닛 생성 (단축 경로) |

**우클릭이 비어 있다**는 점이 이 재설계를 싸게 만든다.

### 1.4b 그리드 단축키 (결정: 2026-07-10)

기존 문자·숫자 단축키(P·B·R·X·1~4·5~7)를 **버린다.** 커맨드 카드의 **물리적 위치**에 대응시킨다:

```
┌───┬───┬───┐
│ Q │ W │ E │
├───┼───┼───┤
│ A │ S │ D │
├───┼───┼───┤
│ Z │ X │ C │
└───┴───┴───┘
```

- 선택 대상이 바뀌어도 **같은 키가 같은 칸**을 누른다. 외울 것이 9개로 고정된다.
- SC2가 도입하고 Legion TD 2가 채택한 방식이다(§1.0).
- 기존 방식(P=생성, 문자 첫 글자 대응)은 SC 원작 방식이지만, 선택 대상마다 카드가 바뀌는
  구조에서는 버튼 위치와 키가 어긋난다.
- `Esc` = 선택 해제 / 하위 메뉴 복귀.

**예시 — 영웅 선택 시:** `Q`힘 `W`민첩 `E`지능 / `A`스킬피해 `S`쿨타임 / (`D`·`Z`·`X`·`C` 빈칸)
**전역(선택 없음):** `Q`프로브 `W`테란 `E`저그 / `A`플토 `S`크리쳐 `D`보스▸ / 하위: `Q`~`D`가 Lv1~6

### 1.5 반응형 — 셀에서 역산한다

첫 초안은 하단 바 높이를 화면 비례(`h * 0.24`)로 잡았다가 **검산에서 폐기했다** —
1080p에서 커맨드 셀이 81px까지 부풀고 HUD 점유율이 27.7%로 지금보다 나빠졌다.
**셀 크기를 먼저 정하고 바 높이를 역산한다:**

```
cell   = clamp(40px, Screen.height * 0.052, 60px)   // 커맨드 카드 한 칸
bottom = cell * 3 + 20                              // 3행 + 패딩
top    = 34                                         // 고정
```

| 해상도 | 하단 바 | HUD 점유 | 보드 높이 | 셀 | 현재 대비 |
|---|---|---|---|---|---|
| 1920×1080 | 188 | 20.6% | 858px | 56px | +2.8%p |
| 1600×900 | 160 | 21.6% | 706px | 47px | −1.9%p |
| 1280×720 | 140 | 24.2% | 546px | 40px | **−9.1%p** |
| 958×385 (실측) | 140 | 45.2% | 211px | 40px | **−29.7%p** |

> **점유율은 지표가 아니다.** 지금의 17.8%는 **전부 보드 위를 덮는다.**
> 새 안의 20.6%는 보드 **바깥**이라 가리는 픽셀이 0이다.
> **완료 판정은 "보드 영역에 오버레이 픽셀 0" + "어떤 해상도에서도 패널이 화면 밖으로 나가지 않음"이다.**

- 세로 385px 같은 극단 창에서는 보드가 211px밖에 안 남는다. **최소 높이 가드**가 필요하다 —
  `Screen.height < 500`이면 커맨드 카드를 3×3에서 3×2로 접고 상세 열을 숨긴다.

## 2. 단계

각 단계는 독립 커밋 가능하고, 그 단계만으로 화면이 나아진다.

#### H1. 선택 모델 + 입력 재편 · 규모 M · **코드 완료, 런타임 검증 대기**
- `View/Selection.cs` 신설 — `None | Hero | Tower | EmptyTile`. Core 무수정.
  타워 선택 시에만 `Game.Selected`를 동기화한다(`SellSelected()`가 그걸 본다).
- `View/CommandCard.cs` 신설 — 3×3 커맨드를 **데이터로** 정의. 키보드와 버튼이 같은 경로를 탄다.
  H2·H3는 이걸 그리기만 하면 된다.
- 영웅에 콜라이더 복원(`GameView.cs:589`의 `Destroy` 제거) + `HeroMarker`.
- 좌클릭 = 선택, 우클릭 = 영웅 이동, Esc = 해제/뒤로. 구 단축키 P·B·R·X·1~7 제거.
- HUD는 임시로 카드를 세로로 펼쳐 보여준다 — H2에서 하단 바로 대체.
- **완료 판정:** ✅ 타입체크 0에러(View·Core). ⏳ **에디터에서 영웅·타워·빈 타일 선택과
  우클릭 이동을 실제로 확인해야 한다** — 타입체크는 클릭이 되는지 알려주지 않는다.

#### H2. 하단 커맨드 바로 통합 · 규모 L · 의존 H1
- `heroPanel` + `messageBar` + `rightPanel` → 하단 바 하나.
- 선택 대상별 3분할 렌더(§1.3). 반응형 높이(§1.5).
- **완료 판정:** 720p·1080p·958×385 세 해상도에서 **패널이 화면을 벗어나지 않고**, 보드 위에 오버레이가 없다.
  각 해상도 스크린샷을 남긴다.

#### H3. 커맨드 카드 3×3 + 단축키 대응 · 규모 M · 의존 H2
- 버튼을 격자 위치에 고정 배치. 단축키를 버튼 위치와 대응.
- 비활성 버튼은 자리를 지키되 흐리게(위치 기억이 근육기억이 되게).
- GOD 타워 타입 변경 버튼의 **자리를 비워둔다**(§4).
- **완료 판정:** 같은 명령이 항상 같은 칸에 있다.

#### H4. 시각 스킨 · 규모 M · 의존 H3 · **완료 (2026-07-12)**
- 한글 폰트(상위 계획 P1) 적용, 기능군별 색, hover/press 상태, 자원 아이콘.
- **선행:** 상위 계획의 P1(폰트)이 끝나 있어야 한다.
- 결과: '모던 다크 + 테크 액센트' — chamfer 형태 · 유리 재질 · 청록 크롬 · SDF 아이콘 16종 ·
  Rajdhani+Pretendard 폴백. 상세는 진행 기록 참조. 관련 파일:
  `UiTheme.cs`(토큰·스프라이트 팩토리) · `UiKit.cs`(유리판 조립) · `HudIcons.cs`(아이콘) ·
  `CommandCard.cs`(칸별 글리프) · `GameHud.cs`(배치).

#### H5. 프로덕션 HUD 업그레이드 · 규모 XL · 의존 H1~H4 · **계획 (2026-07-16)**

H4의 코드 생성 스킨은 기능 검증용 프로토타입으로는 성공했지만 완성 HUD 기준에는 미달한다.
청록 네온을 중세 팔레트로 바꾼 것만으로는 모든 요소가 같은 각진 사각형인 구조적 문제가
해결되지 않았다. H5는 **색 교체가 아니라 형태·재질·정보 위계·피드백을 다시 설계**한다.

상세 실행안과 검증 게이트는 §7을 따른다.

## 3. 구현 방식 — uGUI + TextMeshPro (2026-07-12 갱신)

H1~H3 구조 검증까지는 OnGUI로 진행했고, H4(시각 스킨)에서 예정대로 재검토한 결과
**uGUI + TMP로 이관**을 채택한다:

- OnGUI의 상한은 구조적이다 — 트랜지션·transform·그림자·블러가 없어 "주스"
  (버튼 반동, 카운트업, 슬라이드인)를 만들 수 없다. 서비스 게임 체감의 핵심이 빠진다.
- **UI Toolkit이 아닌 이유** (조사 완료): `box-shadow`·`backdrop-filter` 미지원 —
  떠 있는 패널·글래스가 네이티브로 안 된다. 런타임 게임 HUD로는 설익었다.
- uGUI는 이 게임의 요구와 정합: 9-slice 프레임(코드 생성 Sprite로 무에셋 가능),
  TMP 한글 SDF(빌드 한글 깨짐 급소 동시 해결), 월드 스페이스 캔버스(HP바·플로팅 텍스트),
  트윈. 에셋 0 원칙은 이미 폐기 — 폰트 .ttf 커밋.
- 구현: `TMP_FontAsset.CreateFontAsset(폰트)` 런타임 생성(동적 아틀라스) —
  에디터 Font Asset Creator 불필요. UI 계층도 코드로 조립(프리팹 없음).
- 순서: ① TMP 한글 폰트 ② 하단 커맨드 바 uGUI 재구현(§1 레이아웃) ③ 오버레이(스탯·증강·게임오버)
  ④ 주스(트윈·카운트업). 기존 CommandCard.cs(데이터)는 그대로 — 그리는 쪽만 교체.

## 7. H5 — 프로덕션 HUD 업그레이드 계획

### 7.1 현재 진단

2026-07-16 플레이 화면 검토에서 다음 문제가 확인됐다.

1. **모든 요소가 같은 chamfer 사각형이다.** 상단 상태판, 선택 정보, 커맨드 버튼, 보스 소환이
   색과 크기만 다르고 실루엣이 같다. 기능 위계가 형태로 읽히지 않는다.
2. **재질이 색상값에 머문다.** `UiTheme`의 단색과 런타임 생성 Sprite만으로 철·목재·양피지의
   표면 차이를 만들 수 없다. 중세 팔레트여도 개발자 도구처럼 보인다.
3. **텍스트와 작은 아이콘이 같은 밀도로 반복된다.** 전투 중 중요한 생명·공세·비용과
   보조 정보인 점수·누적 피해가 비슷한 무게로 노출된다.
4. **커맨드 카드가 빈 소켓까지 강조한다.** 3×3 근육기억은 유효하지만, 실제 행동보다 격자
   자체가 먼저 보인다.
5. **보스 단계 6칸의 상태 구분이 약하다.** 직접 단계 선택은 구현됐으나 해금·봉인·쿨타임·
   최고 위험 단계가 작은 글자와 명도 차이에 의존한다.
6. **피드백이 배경색 전환과 축소에 치우친다.** 비용 부족, 해금, 자원 획득, 생명 감소가 서로
   다른 사건으로 느껴지지 않는다.

### 7.2 목표와 비목표

**목표**

- 1초 안에 `생명 → 다음 공세 → 금화/마정석 → 현재 가능한 행동` 순으로 찾을 수 있다.
- 패널 실루엣만 보고 상단 상태, 선택 정보, 전투 명령, 보스 소환을 구분한다.
- 스타일라이즈드 로우폴리 중세 판타지 월드와 같은 철·목재·양피지·황동 재질 언어를 쓴다.
- 960×540에서도 버튼 라벨과 비용이 잘리지 않고, 16:9 고해상도에서 과도하게 커지지 않는다.
- 입력과 게임 규칙은 유지한다. UI 업그레이드가 Core 밸런스를 바꾸지 않는다.

**비목표**

- 장식용 왕국사·문장 설정을 새로 확정하지 않는다.
- 미니맵, 영웅 수동 스킬 버튼, 새 자원, 새 보스 규칙을 추가하지 않는다.
- 웹 HUD를 동시에 개편하지 않는다.
- 첫 단계부터 완성 아트를 대량 제작하지 않는다. 흑백 구조 검증을 먼저 통과한다.

### 7.3 목표 레이아웃

```text
┌ 왕국 표식 · 공세 ───────── 자원 벨트 ───────── 생명 · 점수 ┐
│                                                            │
│                         전투 화면                           │
│                                                            │
│ ┌ 방패형 선택 정보 ┐              ┌ 붉은 보스 봉인 두루마리 ┐ │
│ └─────────────────┘              ├────────────────────────┤ │
│                                  │ 목재 전투 명령 슬롯      │ │
│                                  └────────────────────────┘ │
└────────────────────────────────────────────────────────────┘
```

역할별 형태 언어:

| 영역 | 실루엣 | 주재질 | 우선 정보 |
|---|---|---|---|
| 상단 상태 | 얇은 금속 리본 3개 | 흑철 + 황동 리벳 | 생명, 공세, 자원 |
| 선택 정보 | 방패/문장패 | 흑철 + 가죽 | 이름, HP/XP, 핵심 능력치 |
| 커맨드 카드 | 정사각 목재 슬롯 | 목재 + 철제 모서리 | 아이콘, 비용, 단축키 |
| 보스 소환 | 긴 봉인 두루마리 | 양피지 + 붉은 밀랍 + 사슬 | I~VI, 봉인, 쿨타임 |
| 증강 | 독립 카드 | 양피지 + 등급 인장 | 이름, 효과, 등급 |
| 게임오버 | 중앙 대형 인장 | 흑철 + 왕실 황동 | 점수, 결과, 재시작 |

### 7.4 최소 UI 아트 팩

런타임 단색 Sprite를 완성 자산으로 간주하지 않는다. 아래 최소 세트를 9-slice 중심으로 만든다.

- 흑철 패널 9-slice: 기본/위험 2종
- 목재 슬롯 9-slice: 기본/hover/pressed/disabled 4상태
- 양피지 패널 9-slice: 일반/붉은 보스 2종
- 황동 외곽선·코너·리벳 장식
- 원형 자원 아이콘 베젤
- HP/XP/쿨타임 게이지 프레임과 채움 텍스처
- 보스용 밀랍 봉인, 쇠사슬, 자물쇠
- 선택 대상 초상 또는 실루엣 프레임

자산 경로 원칙:

```text
unity/Assets/Art/UI/Medieval/
├── Frames/
├── Buttons/
├── Bars/
├── Boss/
└── Icons/
```

파일명은 영문 kebab-case, import texture type은 Sprite(2D and UI), 패널은 Sprite Editor에서
border를 설정한다. 동일 기능의 상태 이미지는 크기와 border를 동일하게 유지한다.

### 7.5 정보 위계

**1순위 — 항상 크게:** 생명, 현재 공세, 다음 공세 시간, 금화, 마정석.

**2순위 — 행동 맥락에서 크게:** 선택 대상 HP, 행동 아이콘, 비용, 보스 해금 상태.

**3순위 — 작게:** 점수, 킬, 프로브 수, 누적 피해, 세부 능력치.

선택 정보는 텍스트 문장 대신 다음 구조로 제한한다.

- 좌측: 초상/실루엣
- 중앙 상단: 이름·등급
- 중앙: HP/XP 또는 HP
- 하단: 공격력·DPS·사거리 3개 핵심 수치
- 증강·태그: 작은 인장 아이콘, 상세는 툴팁

### 7.6 커맨드 카드 규칙

- 3×3 위치와 QWE/ASD/ZXC 대응은 유지한다.
- 빈 슬롯은 프레임만 20~30% 명도로 남긴다.
- 행동 이름보다 아이콘을 크게, 비용은 우하단 자원 배지로 표시한다.
- 단축키는 좌상단 금속 리벳/키캡으로 표시한다.
- 비활성 상태를 `비용 부족 / 잠금 / 쿨타임`으로 구분한다.
- hover 시 선택 정보 패널 옆에 1개의 공용 툴팁을 띄운다.
- press는 단순 scale 변화가 아니라 눌림 깊이, 짧은 광택, 기능군별 소리를 함께 사용한다.

### 7.7 보스 단계 소환 규칙

- 큰 단일 소환 버튼은 되돌리지 않는다. `I~VI` 직접 선택을 유지한다.
- 패널 자체가 보스 기능임을 말하므로 각 단계에 같은 보스 아이콘을 반복하지 않는다.
- 해금 단계는 붉은 밀랍 봉인, 잠긴 단계는 쇠사슬/자물쇠로 구분한다.
- 쿨타임은 각 칸의 작은 숫자가 아니라 소환대 전체를 덮는 원형 또는 수평 봉인 게이지로 표현한다.
- 소환 가능한 최고 단계만 약하게 맥동한다. 저단계 버튼은 안정적으로 유지한다.
- hover 툴팁에 단계, 처치 보상, 해금 조건, 소환 가능 여부를 표시한다.

### 7.8 피드백·모션·사운드

| 사건 | 시각 피드백 | 사운드 방향 |
|---|---|---|
| 자원 획득 | 숫자 카운트업 + 베젤 1회 광택 | 동전/수정 짧은 음 |
| 비용 부족 | 부족 자원 배지만 좌우 흔들림 | 마른 금속 클릭 |
| 버튼 hover | 황동 테두리 광택 이동 | 없음 또는 매우 약한 가죽음 |
| 버튼 press | 2~3px 눌림 + 즉시 복귀 | 기능군별 클릭 |
| 보스 해금 | 사슬 파열 + 밀랍 봉인 점등 | 낮은 금속 파열음 |
| 생명 감소 | 생명판 충격 + 짧은 적색 비네트 | 둔중한 타격음 |
| 공세 시작 | 상단 전투 깃발 0.6~0.9초 | 북/뿔피리 짧은 음 |

반복 이벤트는 과도하게 누적되지 않도록 동시에 하나만 재생하거나 강도를 합산한다.
`Time.unscaledDeltaTime`을 사용해 일시정지 오버레이에서도 UI가 멈추지 않게 한다.

### 7.9 구현 단계와 게이트

#### H5-0. 기준 캡처와 계측 · 규모 S

- 960×540, 1280×720, 1920×1080에서 `None/Hero/Tower/EmptyTile` 상태 캡처.
- HUD가 가리는 화면 비율, 최소 글자 크기, 버튼 실제 픽셀 크기를 기록한다.
- **게이트:** 비교 기준 이미지와 수치가 없으면 다음 단계로 가지 않는다.

#### H5-1. 흑백 와이어프레임 · 규모 M

- 실제 아트 없이 역할별 실루엣과 정보 위계만 교체한다.
- 상단 리본, 방패형 정보판, 목재 슬롯, 보스 두루마리의 크기·위치를 검증한다.
- **게이트:** 1초 탐색 테스트에서 생명/공세/두 자원을 모두 찾고, 960×540 잘림이 없다.

#### H5-2. UI 아트 팩 제작 및 9-slice 적용 · 규모 L

- §7.4 최소 자산을 제작하고 `UiTheme`가 Sprite 참조를 제공하도록 바꾼다.
- `UiKit`은 런타임 도형 생성 대신 9-slice 조립을 기본 경로로 사용한다.
- 자산 누락 시 현재 코드 생성 Sprite를 개발용 폴백으로 남긴다.
- **게이트:** 패널을 흑백으로 보아도 역할별 실루엣과 재질이 구분된다.

#### H5-3. 커맨드·보스 상태 재설계 · 규모 L

- 비용 배지, 상태별 비활성 표현, 공용 툴팁을 추가한다.
- 보스 I~VI를 밀랍/사슬 상태로 교체하고 공용 쿨타임 게이지를 추가한다.
- **게이트:** 색을 제거한 캡처에서도 사용 가능/잠금/쿨타임을 구분한다.

#### H5-4. 선택 정보 압축 · 규모 M

- 초상 프레임, 핵심 수치 3개, 태그 인장으로 재배치한다.
- 긴 문장과 누적 피해는 툴팁 또는 `None` 상태 요약으로 옮긴다.
- **게이트:** 영웅/타워/적을 선택했을 때 이름·HP·역할을 1초 안에 읽는다.

#### H5-5. 모션·사운드 · 규모 M

- §7.8 이벤트를 우선순위대로 추가한다.
- 애니메이션 중 입력 차단, 이벤트 중첩, 일시정지 동작을 검사한다.
- **게이트:** 10분 플레이에서 반복 피드백이 방해된다는 관찰이 없어야 한다.

#### H5-6. 해상도·플레이테스트 마감 · 규모 M

- 세 기준 해상도와 극단 창 비율에서 캡처 회귀 검증.
- 신규 플레이어 3명 이상에게 1초 탐색 및 보스 단계 선택 과제를 준다.
- 실패한 요소만 수정하고 장식을 추가하기 위한 수정은 하지 않는다.

## 7.12 그래픽 자산 프로덕션 실행안 (상용 수준) — 2026-07-16 보강

§7.4가 "무엇이 필요한가"의 목록이라면, 이 절은 **"어떤 파일을 어떤 규격으로 만들어 어떻게 코드에
물릴까"**의 실행안이다. 게임 로직·레이아웃·입력은 건드리지 않고 **그래픽 레이어만** 교체한다.

### 7.12.0 현재 상태 — 왜 이미지 리소스가 필요한가

현행 HUD는 **이미지 에셋이 0**이다. 모든 판·버튼·게이지·아이콘이 `UiTheme.cs`의 부호거리장에서
런타임 생성된다(`Chamfer`·`InnerGlow`·`Scanlines`·`Shadow`, `HudIcons.cs`의 SDF 16종). 이 방식은
해상도·팔레트 독립이라 프로토타입엔 이상적이지만 **단색 알파 마스크라 재질(철·목재·양피지·황동)을
표현할 수 없다.** 색을 중세 팔레트로 바꿔도 "각진 개발자 UI"가 남는 근본 원인이다(§7.1 진단).

→ 상용 수준의 갈림길은 **재질을 담은 9-slice 프레임 이미지**를 만들어 런타임 도형을 대체하는 것이다.
코드 생성은 **에셋 누락 시 개발용 폴백**으로만 남긴다(§7.9 H5-2 게이트와 동일).

### 7.12.1 자산 사양표 (제작 작업지시)

원본은 **2배 해상도(@2x)로 저작**하고 캔버스 스케일러 기준 1920×1080에서 1x로 표시한다.
9-slice `border`는 원본(@2x) 픽셀 기준. PPU 100 고정. 전부 `RGBA32`, **sRGB**, 알파 포함.

| 자산 | 역할 실루엣 | 원본(@2x) | 9-slice border | 상태 | 개수 |
|---|---|---|---|---|---|
| `panel-iron` | 상단 리본·정보판 흑철 프레임 | 128×128 | 28 | 기본·위험(붉은 테) | 2 |
| `slot-wood` | 커맨드 카드 목재 슬롯 | 96×96 | 22 | 기본·hover·press·disabled | 4 |
| `parchment` | 보스 소환대·증강 카드 양피지 | 160×96 | 30 | 일반·보스(붉은 밀랍) | 2 |
| `bezel-round` | 자원 아이콘 원형 황동 베젤 | 64×64 | — (원형, 9-slice 아님) | 기본 | 1 |
| `gauge-frame` | HP/XP/쿨타임 게이지 외곽 | 128×32 | 좌우 14 | 기본 | 1 |
| `gauge-fill` | 게이지 채움 텍스처(가로 타일) | 32×32 | 좌우 8 | HP·XP·쿨타임(색만 틴트) | 1 |

과잉 제작 금지 — 이 표를 넘는 장식(배경 일러스트, 프레임별 고유 문양)은 H5-6 통과 후 별건.

### 7.12.2 소싱 전략 — CC0 제약 하에서 (worldbuilding §8)

이 저장소는 공개라 **커밋 자산은 CC0만** 허용된다. 세 경로와 권장:

- **A. CC0 UI 킷 리텍스처 (권장 1순위).** Kenney *UI Pack* 계열(CC0)의 9-slice 프레임/버튼을
  기반으로 색·모서리 장식만 우리 팔레트로 덧입힌다. 라이선스가 깨끗하고 9-slice border가 이미
  잡혀 있어 제작비가 가장 싸다. 중세 목재/철 룩은 Kenney *Fantasy UI*·*RPG UI* 계열이 근접.
- **B. 절차적 저작 + 재질 오버레이 (권장 2순위·병행).** 현행 `Chamfer` 실루엣을 베이스 알파로
  쓰고, 그 위에 **타일 노이즈(철 그레인)·목재 결·양피지 얼룩** 텍스처(CC0 또는 코드 생성)를 곱해
  재질을 입힌다. 이미지 1~2장(노이즈 아틀라스)만 커밋하면 되고 팔레트 독립을 일부 유지한다.
- **C. 생성형 AI로 콘셉트 → 정리.** worldbuilding 아트 DNA(§7.2~7.3) 프롬프트로 프레임 시안을
  뽑고 9-slice로 손 정리. **주의: 생성물의 라이선스·재배포 조건을 확정할 수 있을 때만 커밋**
  (불확실하면 A/B로 대체). 생성 프롬프트·원본은 `resources/`에서 관리(worldbuilding §7 규칙).

권장 조합: **A(프레임 골격) + B(재질 오버레이)**. C는 아이콘/봉인 같은 단발 장식에 한정.

### 7.12.3 9-slice 저작 규격

- **모서리(corner)는 고정, 가운데(center)만 늘어난다.** 원본에서 border 안쪽 코너 영역에 리벳·
  절단·장식을 몰아넣고, 상하/좌우 변은 **한 방향 타일**, 중앙은 **평평한 재질**만 둔다. 코너에
  방향성 무늬가 있으면 늘렸을 때 뭉개진다.
- **모서리 절단(chamfer) 룩 유지:** 코너 픽셀의 바깥 삼각형을 투명 처리해 45° 방패 실루엣을 낸다
  (현행 `ChamferDist`와 같은 형태 언어).
- **상태 이미지는 border·크기·앵커를 완전히 동일하게.** hover/press/disabled가 어긋나면 버튼이
  튄다(§7.4 규칙 재확인).
- **Import 세팅:** Texture Type = `Sprite (2D and UI)`, Mesh Type = `Full Rect`, Wrap = `Clamp`
  (게이지 채움만 `Repeat`), Filter = `Bilinear`, Compression = `High Quality`(작아서 용량 무해),
  sRGB 체크, `Sprite Editor`에서 border 4변 입력. 게이지 채움은 좌우 border만.

### 7.12.4 코드 통합 훅 — 런타임 폴백을 유지한 채 Sprite 주입

**경로 재조정 (중요):** §7.4는 `unity/Assets/Art/UI/Medieval/`를 제시하지만, 이 HUD는 프리팹·
인스펙터가 없고 전부 `Resources.Load`로 조립된다. 따라서 자산은 **`unity/Assets/Resources/UI/Medieval/`**
아래 둬야 코드에서 로드된다(또는 부트스트랩에서 `SpriteAtlas` 하나를 프리로드). 폴더 구조:

```text
unity/Assets/Resources/UI/Medieval/
├── frames/   (panel-iron*, parchment*, portrait-frame*)
├── buttons/  (slot-wood*)
├── bars/     (gauge-frame, gauge-fill)
├── boss/     (boss-seal-wax, boss-seal-lock, boss-chain)
└── deco/     (bezel-round, corner-brass)
```

`UiTheme.cs`에 **역할→Sprite 로더**를 추가한다. 있으면 authored 9-slice, 없으면 현행 `Chamfer`로 폴백:

```csharp
// UiTheme.cs (신설) — 승격 게이트. 에셋이 없으면 지금 그대로 코드 생성으로 돈다.
public enum Frame { PanelIron, PanelIronDanger, Slot, SlotHover, SlotPress, SlotDisabled,
                    Parchment, ParchmentBoss, PortraitFrame }
static readonly Dictionary<Frame, Sprite> frameCache = new();
public static Sprite FrameSprite(Frame f)
{
    if (frameCache.TryGetValue(f, out var s)) return s;
    s = Resources.Load<Sprite>("UI/Medieval/" + PathOf(f));   // authored 9-slice
    if (s == null) s = Chamfer(CutFor(f), StrokeFor(f));       // ← 개발용 폴백(현행 유지)
    frameCache[f] = s;
    return s;
}
```

그리고 `UiKit.cs`의 조립 함수(`GlassPlate`·`Panel`·`FloatingPanel`·`Button`·`Bar`)가 `UiTheme.Chamfer`를
직접 부르던 자리를 `UiTheme.FrameSprite(역할)`로 바꾼다. **Image.type을 `Sliced`로** 설정하고
(현재 일부 `Simple`) `pixelsPerUnitMultiplier`로 border 스케일을 잡는다. 호출부 시그니처는 유지 —
그리는 스프라이트만 교체되므로 `GameHud.cs` 레이아웃 코드는 무변경.

이 훅 하나로 "에셋을 채우는 만큼 점진적으로" 상용화된다. 한 자산을 넣으면 그 역할만 authored로
바뀌고 나머지는 폴백. H5-2 게이트("흑백 실루엣으로 역할 구분")를 자산별로 통과시키며 진행.

### 7.12.5 타이포그래피 승격 (그래픽 구멍 — 우선순위 높음)

현행 주 폰트는 **Rajdhani(SemiBold/Bold) — 테크 디스플레이체**다. 이는 H4의 "모던 다크+테크"
스킨 잔재로, **중세 전장 시각 언어(§7.3)와 정면으로 충돌**한다. 숫자가 SF 계기판처럼 찍힌다.
상용 수준에선 폰트가 룩의 절반이다. 교체안:

- **제목·라벨(디스플레이):** 중세감의 **세리프 또는 반(半)블랙레터** — OFL/CC0 중 `Cinzel`,
  `IM Fell`, `MedievalSharp` 계열. 라틴·숫자용. (블랙레터 원형은 가독성이 낮아 제목 한정.)
- **수치(HUD 계기):** 표 정렬(tabular figures) 되는 세리프/슬래브 — 자원·비용·라이프가 흔들리지
  않게. 위 디스플레이 폰트의 tnum 피처 또는 별도 슬래브.
- **한글 본문:** `PretendardKR` 유지(가독성). 단, 제목 한글은 명조 계열(예 `Nanum Myeongjo`,
  OFL) 폴백을 얹어 라틴 디스플레이와 톤을 맞춘다.
- **적용점:** `UiTheme.Font`는 이미 주+폴백 구조라, `Rajdhani` 로드를 디스플레이 폰트로 교체하고
  제목/수치/본문을 **FontLabel/Title별로 다른 FontAsset**을 반환하도록 분리한다(현재 단일 반환).
  라이선스는 `Resources/Fonts/LICENSE-*.txt` 패턴 그대로 커밋.

### 7.12.6 아이콘 세트 승격

`HudIcons.cs`의 16종 SDF는 실루엣 골격으론 충분하나 상용에선 **일관 그리드·2톤 음각(engraved
metal) 스타일**의 authored 아이콘이 낫다. 규격: 원본 **64×64 @2x, 32px 세이프존, 2~3px 외곽선,
단색+음영 1단**. `HudIcons.Get(id)`가 `Resources.Load<Sprite>("UI/Medieval/icons/{id}")` 우선,
없으면 현행 런타임 SDF 폴백 — §7.12.4와 동일 패턴. 자원 아이콘은 §7.12.1 `bezel-round` 안에 얹어
동전/수정/두루마리 실물감을 준다.

### 7.12.7 게이지·바 프레임 (3레이어)

HP/XP/쿨타임을 단색 두 쿼드에서 **프레임+채움+광택 3레이어**로 승격:

1. `gauge-frame` 9-slice(황동 테) — 최하단.
2. `gauge-fill` 타일 채움 — `Image.type=Filled`(Horizontal) + 색 틴트(HP=이끼/XP=금/쿨=강철).
3. (선택) 상단 **광택 스트립** 반투명 오버레이 — 유리/에나멜 하이라이트.

월드 HP바(`GameViewFx.MakeBar`)와는 별개다 — 그건 3D 빌보드, 이건 HUD uGUI. 재질 톤만 맞춘다.

### 7.12.8 해상도·아틀라스·성능

- **단일 `SpriteAtlas`**(`ui-medieval.spriteatlas`)로 위 자산을 묶어 드로우콜을 합친다. 부트스트랩
  에서 `atlas.GetSprite(name)`로 접근하거나 Resources 병행.
- 원본 @2x 저작 → 1080p 1x 표시. Canvas Scaler `Scale With Screen Size`, 기준 1920×1080,
  match 0.5. 960×540에서도 border가 뭉치지 않게 border는 "표시 최소 패널 크기의 절반 미만"으로.
- 텍스처는 작아 압축 없이 `RGBA32`로 둬도 무해(<수백 KB 합계). 밉맵 끔(UI).

### 7.12.9 라이선스·프로비넌스

자산마다 출처·라이선스를 `resources/README.md`(또는 `resources/ui-medieval/SOURCES.md`)에 기록:
원본 킷 이름·URL·라이선스(CC0 확인)·수정 내역. 생성형 AI 산출물은 프롬프트와 원본을 함께 남긴다.
**CC0 미확인 자산은 커밋하지 않는다**(worldbuilding §8). 완료 기준 §7.10의 "라이선스·원본 위치
기록"을 이 파일로 충족한다.

### 7.12.10 제작 순서와 자산별 게이트

§7.9 H5-2에 매핑되는 그래픽 전용 세부 순서:

1. **폰트 교체 먼저** (§7.12.5) — 코드 두 줄, 체감 변화 가장 큼. 룩의 절반이 여기서 바뀐다.
2. `panel-iron` + `slot-wood` 9-slice 도입 → `UiTheme.FrameSprite` 훅 연결(§7.12.4).
   **게이트:** 흑백 캡처에서 상단 리본 / 정보판 / 커맨드 슬롯 실루엣·재질이 구분된다.
3. `gauge-frame/fill`, `bezel-round`, 아이콘 세트.
4. `parchment` + `boss-seal/chain` — 보스 소환대(§7.7).
   **게이트:** 색 제거 캡처에서 해금(밀랍)·잠금(사슬)·쿨타임(게이지)이 구분된다.
5. `corner-brass`·`portrait-frame` 마감.

각 단계는 **9-slice 스트레치 테스트**(최소~최대 패널 크기에서 코너 뭉갬·이음매 확인)와
**흑백 실루엣 테스트**를 통과해야 다음으로 간다. 자산이 폴백보다 나쁘면 폴백을 유지한다.
